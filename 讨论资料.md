## 🔍 **问题分析：为什么会出现这些问题？**

### **问题1：标题与核心观点矛盾**

**根源**：

- 初始框架的标题是"从C端到B端"
- 但经过三次反思，核心观点变成了"B端是路径，C端是目标"
- **我在生成Cursor指令时，忘记同步更新标题**

**教训**：

- ⚠️ **标题是核心观点的浓缩，必须与内容立场一致**
- ⚠️ **每次全局反思后，必须检查标题是否需要调整**

------

### **问题2：失败案例缺少细节**

**根源**：

- 我在Cursor指令中写了"要有具体案例"
- 但**没有给出案例的具体内容**（只是要求）
- Cursor只能生成简略版（缺少时间线、对话、反思）

**教训**：

- ⚠️ **对于"高冲击力"的内容（如失败案例），不能只给要求，要给完整文字**
- ⚠️ **Claude应该先写关键案例，再交给Cursor整合**

------

### **问题3：格式和编号错误**

**根源**：

- Cursor的小疏漏（###3.2缺少空格，7.3编号重复）

**教训**：

- ⚠️ **任何工具都会犯小错误，质检清单必须包含格式检查**

------

### **问题4：缺少YAML索引**

**根源**：

- 我要求Cursor生成YAML，但Cursor可能：
  - 不确定doc_id（需要查看现有编号）
  - 不确定YAML格式（虽然我给了模板）

**教训**：

- ⚠️ **对于需要"上下文信息"的内容（如doc_id），Claude应该直接生成，而不是让Cursor猜**

## 🔄 **六阶段写作流程**

### **阶段1：设计框架（Claude主导）**

**输入**：

- 用户的初始想法或问题

**工作流**：

```
Step 1: 搜索知识库
→ 用project_knowledge_search找相关文档
→ 识别理论空白

Step 2: 设计框架
→ 提出2-3个框架选项
→ 用户选择方向

Step 3: 案例准备
→ 列出需要的案例清单
→ 明确每个案例的作用（主线/对比/失败）
```

**输出**：

- 完整框架大纲
- 案例清单
- 理论链条

**关键检查**：

-  框架与用户需求一致吗？
-  案例足够支撑框架吗？
-  理论链条完整吗？

------

### **阶段2：三次反思机制（Claude主导）**

**为什么是三次？**

- **第一次反思（25%进度）**：立场检查
- **第二次反思（50%进度）**：案例检查
- **第三次反思（75%进度）**：深度检查

------

#### **反思1：立场检查（在写第二章前）**

**检查清单**：

```
□ 文章的核心观点是什么？
□ 开场故事是否传递了正确的观点？
□ 有没有偏向某一方（如"C端错B端对"）？
□ 标题是否与核心观点一致？
```

**如果检查失败**：

- 立即调整立场
- **同步更新标题**（这次我们犯的错误）
- 重写已完成的部分

------

#### **反思2：案例检查（在写第三章前）**

**检查清单**：

```
□ 主线案例是否完整？
□ 是否需要对比案例？
□ 是否需要失败案例？
□ 案例是否覆盖所有战略/场景？
```

**如果检查失败**：

- 补充案例
- 调整框架（如我们扩展为"三种战略"）

------

#### **反思3：深度检查（在写最后1/4前）**

**检查清单**：

```
□ 文章有没有"反常识"的冲击？
□ 有没有"温水煮青蛙"式的失败案例？
□ 有没有"道"的阐释（不只是术）？
□ 读者能否产生"啊哈时刻"？
```

**如果检查失败**：

- 增加失败案例的细节
- 增加"底层逻辑"章节
- 提升理论深度

------

### **阶段3：关键内容生成（Claude主导）**

**什么内容必须由Claude生成？**

| 内容类型         | 为什么必须Claude生成                 | 反例（失败教训）             |
| ---------------- | ------------------------------------ | ---------------------------- |
| **失败案例细节** | 需要"温水煮青蛙"的时间线、对话、反思 | 本次：Cursor生成的案例太简略 |
| **核心金句**     | 需要提炼精髓                         | Cursor可能写成长句           |
| **决策框架**     | 需要逻辑严密                         | Cursor可能遗漏边界条件       |
| **YAML索引**     | 需要知识库上下文（doc_id等）         | 本次：Cursor不知道doc_id     |

**Claude应该输出的"最小完整单元"**：

```
1. 失败案例的完整文字（800-1000字/个）
2. 核心金句的精确措辞（10-30字/条）
3. YAML索引的完整内容
4. 决策矩阵的计算公式和示例
```

------

### **阶段4：Cursor批量生成（Cursor主导）**

**什么内容可以交给Cursor？**

| 内容类型       | 为什么可以交给Cursor         | 前提条件               |
| -------------- | ---------------------------- | ---------------------- |
| **框架性内容** | 基于已有讨论整合             | Claude已经讨论清楚     |
| **案例扩展**   | 基于Claude的主线案例补充细节 | 主线案例已由Claude确定 |
| **理论阐释**   | 基于知识库文档展开           | 知识库有相关内容       |

**Cursor指令的标准格式**：

markdown

~~~markdown
@Codebase 写作任务

## 上下文
你已经阅读了完整对话，了解：
- [关键信息1]
- [关键信息2]

## 任务
按照以下结构生成文档：
[详细结构]

## 关键内容（直接使用，不要改写）
### 失败案例A
[Claude已生成的完整文字]

### 失败案例B
[Claude已生成的完整文字]

### 核心金句
[Claude已生成的10条金句]

### YAML索引
[Claude已生成的完整YAML]

## 注意事项
1. 关键内容直接复制，不要改写
2. 其他内容基于对话生成
3. 格式统一（Markdown标准）
```

**关键原则**：
- ✅ **关键内容由Claude生成，Cursor复制**
- ✅ **框架性内容由Cursor基于对话生成**
- ❌ **不要让Cursor"发挥创造"（容易偏离）**

---

### **阶段5：全局质检（Claude主导）**

**质检时机**：Cursor生成完成后

**质检清单（10项）**：
```
立场一致性（3项）
□ 1. 标题与核心观点一致吗？
□ 2. 开场故事传递的观点正确吗？
□ 3. 文章有没有自相矛盾的地方？

案例完整性（3项）
□ 4. 失败案例有时间线、对话、反思吗？
□ 5. 主线案例在关键章节都出现了吗？
□ 6. 案例与理论框架对应吗？

深度检查（2项）
□ 7. 有没有"反常识"的冲击？
□ 8. 有没有"道"的阐释（不只是术）？

完整性检查（2项）
□ 9. YAML索引完整吗？
□ 10. 相关文档引用正确吗？
~~~

**质检方法**：

- Claude用`view`工具查看Cursor生成的文件
- 对照质检清单逐项检查
- 发现问题后，生成"修复指令"

------

### **阶段6：快速修复（Cursor主导）**

**修复原则**：

- 格式问题：Cursor修复
- 关键内容问题：Claude补充完整内容 → Cursor替换
- 逻辑问题：Claude和用户讨论 → 重新生成

**修复指令格式**：

markdown

~~~markdown
@Codebase 修复文档

## 问题清单
1. [问题1描述]
2. [问题2描述]

## 修复方案
### 修复1：[问题1]
[具体修复步骤]
[如果需要替换内容，给出完整文字]

### 修复2：[问题2]
...

## 验证清单
修复完成后，检查：
- [ ] [验证项1]
- [ ] [验证项2]
```

### **创新1：三次反思机制**

**为什么有效？**
- 25%进度反思立场 → 避免大方向错误

### **创新2：Claude-Cursor分工**

**分工原则**：
```
Claude：
- 思考（框架、反思、核心观点）
- 关键内容（失败案例、金句、YAML）
- 质检（全局检查、修复指令）

Cursor：
- 执行（批量生成、格式整合）
- 基于对话扩展（理论阐释）
- 快速修复（格式、替换）

人类：
- 决策（方向确认、选择方案）
- 验收（最终质量检查）
- 桥梁（在Claude和Cursor之间传递）
```

**为什么这样分工？**
- Claude擅长思考和判断 → 负责核心内容
- Cursor擅长批量生成 → 负责框架整合
- 避免重复劳动 → 节省Token

```

**检验标准**：
- 标题能否一句话概括核心观点？
- 标题有没有歧义？
- 标题有没有与金句呼应？
~~~

~~~markdown
#### 禁止7：标题与核心观点滞后不同步

❌ **失败案例**：
```
对话过程：
- 用户提供框架："从C端到B端：战略转移的底层逻辑"
- 第一次反思：调整立场为"C端是目标，B端是路径"
- 第二次反思：扩展为"三种战略"
- 第三次反思：增加"B端是杠杆，不是归宿"
- 最终交付：标题还是"从C端到B端：战略转移的底层逻辑"

问题：标题暗示"放弃C端"，与核心观点"B端是路径"完全矛盾
```

✅ **成功案例**：
```
对话过程：
- 用户提供框架："从C端到B端：战略转移的底层逻辑"
- 第一次反思：调整立场为"C端是目标，B端是路径"
  → 立即检查标题，发现矛盾
  → 提出3-5个标题选项
  → 用户确认："B端杠杆：撬动C端市场的务实路径"
- 第二次/第三次反思后：再次检查标题，确认一致
- 最终交付：标题与核心观点完美一致

成功原因：每次全局反思后，MUST检查标题
```

**为什么成功**：
- 标题是核心观点的浓缩
- 每次反思 = 观点可能变化 = 必须检查标题
- 用户看标题就能知道文章立场
~~~

# 新文档写作框架指南

> **适用场景**：在知识库中添加新文档时使用  
> **版本**：v1.0（适配道术分层目录结构）  
> **更新时间**：2025-01-20

---

## 📋 快速开始

### 写作新文档的完整流程

```
1. 确定主题 → 2. 判断分类 → 3. 查找空白 → 4. 写作内容 → 5. 创建索引 → 6. 提交仓库
```

---

## 一、确定主题与分类

### 1.1 主题来源

新文档的主题通常来自：

| 来源 | 说明 | 示例 |
|------|------|------|
| **洞察触发** | 商业案例分析、复盘反思 | "发现拼多多的成功不是低价，是改写规则" |
| **理论空白** | 现有理论体系的缺失部分 | "缺少'规则制定者'的系统性框架" |
| **用户需求** | 学员提问、培训反馈 | "如何判断创业项目是否靠谱？" |
| **知识更新** | 新技术、新案例、新理论 | "AI Agent的商业应用案例" |

### 1.2 分类判断框架

**核心问题**：这是"道"还是"术"？

| 维度 | 道层文档 | 术层文档 |
|------|---------|---------|
| **目的** | 建立认知框架 | 解决具体问题 |
| **内容** | WHY（为什么） | HOW（怎么做） |
| **抽象度** | 高（普适原理） | 低（具体方法） |
| **时效性** | 长期有效 | 可能随技术变化 |
| **可操作性** | 思维方式 | 行动步骤 |

**判断流程图**：

```
这篇文档主要解决什么问题？
    ↓
┌────────────────────┬────────────────────┐
│ 改变认知/建立框架   │ 解决具体问题       │
│ （WHY）            │ （HOW）            │
↓                    ↓
道层                  术层
│                    │
↓                    ↓
选择子分类            选择子分类
```

## 二、道层文档分类

### 2.1 Theory（核心理论）

**定义**：建立新的认知框架，解释某个领域的根本运作机制

**适用场景**：
- 提出新的理论模型
- 解释复杂现象的底层机制
- 建立跨学科的认知框架

**典型特征**：
- 有明确的理论模型或框架
- 包含学术引用或科学依据
- 可迁移到多个应用场景

**现有案例**：
- 认知内共生理论
- 心智路径依赖理论
- 双螺旋进化模型

**写作检查清单**：
- [ ] 有明确的理论模型图示
- [ ] 有底层机制的解释
- [ ] 有跨领域的应用示例
- [ ] 有与现有理论的对比
- [ ] 文献或科学依据充分

---

### 2.2 Framework（框架模型）

**定义**：整合多个理论/方法的系统性框架

**适用场景**：
- 整合多个独立理论
- 构建思维模型工具箱
- 建立决策或分析框架

**典型特征**：
- 包含多个子模块
- 有清晰的使用场景
- 可作为"索引"或"地图"

**现有案例**：
- 思想体系核心框架 v1.1

**写作检查清单**：
- [ ] 框架包含3个以上子模块
- [ ] 有清晰的使用说明
- [ ] 有与其他文档的关联
- [ ] 可作为学习路线图
- [ ] 有版本号（便于迭代）

---

### 2.3 Philosophy（人生哲学）

**定义**：关于人生意义、价值观、长期主义的思考

**适用场景**：
- 人生方向选择
- 价值观建立
- 长期主义实践

**典型特征**：
- 有个人故事或深度反思
- 触及人生根本问题
- 跨越具体方法，聚焦意义

**现有案例**：
- 构建北极星
- 时间的朋友
- 人生的意义与实践

**写作检查清单**：
- [ ] 有触动人心的故事
- [ ] 有深度的哲学思考
- [ ] 有可落地的实践建议
- [ ] 有长期主义视角
- [ ] 语言有温度和共鸣

## 四、查找理论空白

在写作前，先查找现有知识库，避免重复。

### 4.1 搜索现有文档

```bash
# 在Project Knowledge中搜索
【搜索】你的主题关键词

# 或在本地搜索
grep -r "关键词" docs/
```

### 4.2 映射理论关系

**问自己三个问题**：

1. **哪些现有理论能解释我的洞察？**
   - 找到相关文档
   - 分析理论覆盖度

2. **现有体系有什么空白？**
   - 识别理论缺失
   - 确认是否需要新文档

3. **新文档与现有文档的关系？**
   - 补充关系（填补空白）
   - 深化关系（展开细节）
   - 对比关系（提供新视角）

### 4.3 确认写作方向

基于映射结果，确定：

- **补充现有文档**：在现有文档中增加章节
- **创建新文档**：理论空白明显，需要独立文档
- **暂不写作**：已有充分覆盖，无需重复

## 五、写作内容（7步法）

### 5.1 标准写作框架

所有文档都应遵循**7步法**：

```
1. 故事/案例开头（必须）
2. 核心洞察/问题（必须）
3. 理论框架/方法（必须）
4. 案例分析/对比（必须）
5. 实践建议/行动（必须）
6. 核心洞察金句（必须）
7. 相关文档引用（必须）
```

---

### 5.2 第一步：故事/案例开头（必须）

**目的**：用故事抓住注意力，建立情境

**要求**：
- 真实案例（公司、个人、历史事件）
- 有冲突或反转
- 与主题直接相关
- 200-500字

**模板**：

```markdown
## 一、[故事标题]

[具体场景描述]

**问题/冲突**：[描述问题]

**结果**：[出人意料的结果]

**这就是本文要讨论的：[核心主题]**
```

**案例**：

```markdown
## 一、那个被"顺手"集成的功能

2018年，微软Teams宣布内置屏幕录制功能。消息一出，许多独立的录屏工具公司股价暴跌。

**为什么？**

因为用户发现：既然Teams已经有了录屏功能，为什么还要单独下载一个录屏工具？

**这就是初创企业面临的残酷现实：你的核心功能，可能只是大平台的"顺手"功能。**
```

---

### 5.3 第二步：核心洞察/问题（必须）

**目的**：提炼本质问题，建立认知冲突

**要求**：
- 一句话点明本质
- 挑战常识或直觉
- 提出清晰问题

**模板**：

```markdown
## 二、核心问题：[问题陈述]

### 传统观点认为：[常识]

### 但真相是：[反常识洞察]

### 这引出一个关键问题：[核心问题]
```

**案例**：

```markdown
## 二、核心问题：创新靠运气还是系统？

### 传统观点认为：创新靠灵感

大多数人认为创新是"灵光一现"，是少数天才的专利。

### 但真相是：创新可以系统化

真正的创新不是等待灵感的降临，而是构建灵感产生的系统。

### 这引出一个关键问题：如何把创新从"靠运气"变成"稳赢"？
```

---

### 5.4 第三步：理论框架/方法（必须）

**目的**：提供可迁移的思维框架或方法

**要求**：
- 有清晰的框架图示
- 有2-5个核心要素
- 有每个要素的详细解释

**模板**：

```markdown
## 三、[框架名称]：核心要素

### 框架概览

[框架图示或表格]

### 要素一：[名称]

**定义**：[清晰定义]

**为什么重要**：[价值说明]

**如何应用**：[具体方法]

**案例**：[实际案例]

### 要素二：[名称]
...
```

**表格示例**：

```markdown
| 要素 | 传统方式 | 新方式 | 关键差异 |
|------|---------|--------|---------|
| 用户端 | 服务高价值用户 | 服务边缘用户 | 改写"谁能买" |
| 分发端 | 位置决定曝光 | 数据决定曝光 | 改写"谁能被看见" |
```

---

### 5.5 第四步：案例分析/对比（必须）

**目的**：用案例验证理论，增强说服力

**要求**：
- 至少3个案例
- 有成功和失败对比
- 有中国和国际案例
- 有数据支撑

**模板**：

```markdown
## 四、案例分析

### 案例1：[成功案例]

**背景**：[公司/场景]

**做对了什么**：[关键决策]

**结果**：[量化结果]

**启示**：[可迁移的经验]

### 案例2：[失败案例]

**背景**：[公司/场景]

**做错了什么**：[致命错误]

**结果**：[惨痛代价]

**启示**：[避坑指南]

### 对比分析

| 维度 | 成功案例 | 失败案例 | 关键差异 |
|------|---------|---------|---------|
```

---

### 5.6 第五步：实践建议/行动（必须）

**目的**：提供可执行的行动步骤

**要求**：
- 分"立即行动"和"长期坚持"
- 每条建议具体可执行
- 有明确的时间、对象、产出

**模板**：

```markdown
## 五、实践建议

### 立即行动（本周）

1. **[行动1]**：
   - 时间：[具体时间]
   - 对象：[具体对象]
   - 产出：[具体成果]

2. **[行动2]**：
   ...

### 长期坚持（本月/季度）

1. **[战略1]**：
   - 目标：[清晰目标]
   - 方法：[具体方法]
   - 检查点：[如何验证]

2. **[战略2]**：
   ...

### 自检问题

- [ ] [关键问题1]
- [ ] [关键问题2]
- [ ] [关键问题3]
```

---

### 5.7 第六步：核心洞察金句（必须）

**目的**：提炼可独立引用的金句

**要求**：
- 至少3条金句
- 每条10-30字
- 可独立理解
- 有冲击力

**模板**：

```markdown
## 六、核心洞察

### 金句

1. **"[金句1]"**

2. **"[金句2]"**

3. **"[金句3]"**

### 核心框架

**[框架名称] = [要素1] + [要素2] + [要素3]**
```

**案例**：

```markdown
### 金句

1. **"改写规则是创新，制定规则是建国。"**

2. **"最强的货币不是黄金，而是信仰。"**

3. **"真正的创新不是等待灵感的降临，而是构建灵感产生的系统。"**
```

---

### 5.8 第七步：相关文档引用（必须）

**目的**：建立文档之间的关联，形成知识网络

**要求**：
- 引用至少2篇相关文档
- 说明关联关系
- 使用标准引用格式

**模板**：

```markdown
---

*与本文相关的文档：*
- *《[文档名]》——[关系说明]*
- *《[文档名]》——[关系说明]*
- *《[文档名]》——[关系说明]*
```

**案例**：

```markdown
---

*与本文相关的文档：*
- *《价值链创新》——如何识别可以改写的规则*
- *《从0到1》——垄断者如何建立护城河*
- *《战略性杠杆》——如何借力建立生态*
- *《时间的朋友》——长期主义视角下的规则制定*
```

## 六、写作风格要求

### 6.1 语气风格

**核心定位**：像一位带徒弟的CEO

**特征**：
- 直白、真诚、有锋芒
- 敢泼冷水，也给路径
- 务实、带节奏感
- 不堆砌概念

**禁止**：
- 学术八股文
- 鸡汤式空洞
- 概念堆砌
- 过度修饰

---

### 6.2 语言要求

| 维度 | 要求 | 示例 |
|------|------|------|
| **简洁** | 每句话<40字 | ✅"改写规则比做得更好更重要"<br>❌"在当前的市场竞争环境下，企业应该..." |
| **具体** | 用数据和案例 | ✅"3年估值10亿"<br>❌"取得了巨大成功" |
| **口语化** | 像说话一样 | ✅"这招狠"<br>❌"该策略具有显著优势" |
| **有画面感** | 用比喻和故事 | ✅"小舢板vs大轮船"<br>❌"资源不对等" |

---

### 6.3 格式要求

**标题层级**：

```markdown
# 一级标题（文档标题）

## 二级标题（章节）

### 三级标题（小节）

#### 四级标题（要点）
```

**列表格式**：

```markdown
# 只在必要时使用列表

**当需要罗列要点时**：
- 要点1
- 要点2

**当需要步骤时**：
1. 步骤1
2. 步骤2
```

**表格格式**：

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容1 | 内容2 | 内容3 |
```

**引用格式**：

```markdown
> **📖 详细内容请参考：**《文档名》——关系说明
```

## 七、创建索引文件

每篇新文档都需要创建对应的YAML索引文件。

### 7.1 索引文件命名

```
rag-index/indexes/DOC-[D/S][编号].yaml
```

**编号规则**：
- 道层：DOC-D013, DOC-D014...
- 术层：DOC-S039, DOC-S040...

**如何确定编号**：

```bash
# 查看当前最大编号
ls rag-index/indexes/ | grep "DOC-D" | sort | tail -1
ls rag-index/indexes/ | grep "DOC-S" | sort | tail -1
```

---

### 7.2 索引文件模板

```yaml
---
doc_id: "DOC-[D/S][编号]"
doc_type: "[theory/methodology/framework/technique/philosophy]"
title: "[文档标题]"
summary: "[文档摘要，100-200字]"
keywords:
  - "[关键词1]"
  - "[关键词2]"
  - "[关键词3]"
related_docs:
  - "[相关文档ID1]"
  - "[相关文档ID2]"
```

## 十一、常见问题

### Q1：如何判断应该创建新文档还是补充现有文档？

**判断标准**：

| 情况 | 操作 |
|------|------|
| 现有文档已充分覆盖 | 不写 |
| 可以作为现有文档的一个章节 | 补充现有文档 |
| 是独立的主题，且内容>2000字 | 创建新文档 |
| 与现有文档有明显差异 | 创建新文档 |

---

```bash
# 方法1：在Claude中搜索
```

# 新文档写作框架指南

> **适用场景**：在知识库中添加新文档时使用  
> **版本**：v2.1（新增案例真实性要求）  
> **更新时间**：2026-02-19

---

## 📋 快速开始

### 写作新文档的完整流程

```
1. 确定主题 → 2. 判断分类 → 3. 查找空白 → 4. 写作内容 → 5. 创建索引 → 6. 提交仓库
```

## 一、确定主题与分类

### 1.1 主题来源

新文档的主题通常来自：

| 来源 | 说明 | 示例 |
|------|------|------|
| **洞察触发** | 理论思考、方法总结、认知提升 | "发现创新不是做得更好，而是做得不同" |
| **理论空白** | 现有理论体系的缺失部分 | "缺少'规则制定者'的系统性框架" |
| **用户需求** | 学员提问、培训反馈 | "如何判断创业项目是否靠谱？" |
| **知识更新** | 新技术、新案例、新理论 | "AI Agent的商业应用案例" |

### 1.2 分类判断框架

**核心问题**：这是"道"还是"术"？

| 维度 | 道层文档 | 术层文档 |
|------|---------|---------|
| **目的** | 建立认知框架 | 解决具体问题 |
| **内容** | WHY（为什么） | HOW（怎么做） |
| **抽象度** | 高（普适原理） | 低（具体方法） |
| **时效性** | 长期有效 | 可能随技术变化 |
| **可操作性** | 思维方式 | 行动步骤 |

**判断流程图**：

```
这篇文档主要解决什么问题？
    ↓
┌────────────────────┬────────────────────┐
│ 改变认知/建立框架   │ 解决具体问题       │
│ （WHY）            │ （HOW）            │
↓                    ↓
道层                  术层
│                    │
↓                    ↓
选择子分类            选择子分类
```

## 二、道层文档分类

### 2.1 Theory（核心理论）

**定义**：建立新的认知框架，解释某个领域的根本运作机制

**适用场景**：
- 提出新的理论模型
- 解释复杂现象的底层机制
- 建立跨学科的认知框架

**典型特征**：
- 有明确的理论模型或框架
- 包含学术引用或科学依据
- 可迁移到多个应用场景

**现有案例**：
- 认知内共生理论
- 心智路径依赖理论
- 双螺旋进化模型

**写作检查清单**：
- [ ] 有明确的理论模型图示
- [ ] 有底层机制的解释
- [ ] 有跨领域的应用示例
- [ ] 有与现有理论的对比
- [ ] 文献或科学依据充分

---

### 2.2 Framework（框架模型）

**定义**：整合多个理论/方法的系统性框架

**适用场景**：
- 整合多个独立理论
- 构建思维模型工具箱
- 建立决策或分析框架

**典型特征**：
- 包含多个子模块
- 有清晰的使用场景
- 可作为"索引"或"地图"

**现有案例**：
- 思想体系核心框架 v1.1

**写作检查清单**：
- [ ] 框架包含3个以上子模块
- [ ] 有清晰的使用说明
- [ ] 有与其他文档的关联
- [ ] 可作为学习路线图
- [ ] 有版本号（便于迭代）

---

### 2.3 Philosophy（人生哲学）

**定义**：关于人生意义、价值观、长期主义的思考

**适用场景**：
- 人生方向选择
- 价值观建立
- 长期主义实践

**典型特征**：
- 有个人故事或深度反思
- 触及人生根本问题
- 跨越具体方法，聚焦意义

**现有案例**：
- 构建北极星
- 时间的朋友
- 人生的意义与实践

**写作检查清单**：
- [ ] 有触动人心的故事
- [ ] 有深度的哲学思考
- [ ] 有可落地的实践建议
- [ ] 有长期主义视角
- [ ] 语言有温度和共鸣

## 四、写作过程管理

### 4.1 三次反思检查点（MUST执行）

理论文档写作是"观点迭代"的过程，初始框架可能不完美。通过三次反思，在写作过程中不断校准立场、补充案例、提升深度。

#### 反思检查点1：立场一致性检查（完成25%进度时）

**触发时机**：完成第1-2章后（或总字数达到25%时）

**检查对象**：标题、开场故事、核心观点

**立场一致性清单**：

- [ ] **标题与核心观点一致吗？**
  - 检查方法：标题能否一句话概括文章立场？
  - 常见问题："从A到B"暗示放弃A，但正文说"B是路径，A是目标"
  
- [ ] **开场故事传递的观点正确吗？**
  - 检查方法：故事结尾的洞察与核心观点一致吗？
  - 常见问题：故事传递"A错B对"，但正文说"不同阶段不同选择"
  
- [ ] **有没有偏向某一方？**
  - 检查方法：是否过度强调某个方案的优势？
  - 常见问题：全文都在说B端好，没提B端的代价

- [ ] **如果立场需要调整 → 立即调整 → MUST同步更新标题**

**处理流程**：
1. 如果发现立场问题 → 停止写作
2. 调整立场（修改已完成部分）
3. 提出3-5个新标题选项
4. 用户确认后继续

---

#### 反思检查点2：案例完整性检查（完成50%进度时）

**触发时机**：完成第3-4章后（或总字数达到50%时）

**检查对象**：主线案例、对比案例、失败案例

**案例完整性清单**：

- [ ] **主线案例是否完整？**
  - 完整标准：有背景、有问题、有解决方案、有结果
  - 检查方法：主线案例是否在多个章节出现（如第1/3/6章）？
  
- [ ] **是否需要对比案例？**
  - 需要条件：如果提出"三种战略/方法"，每种都要有案例
  - 检查方法：每个理论框架是否有案例支撑？
  
- [ ] **是否需要失败案例？**
  - 需要条件：如果有"警示/陷阱"章节，必须有失败案例
  - 检查方法：失败案例是否有"温水煮青蛙"的过程细节？
  
- [ ] **案例是否覆盖所有战略/场景？**
  - 检查方法：列出框架中的所有方法，每个方法是否有案例？

**处理流程**：
1. 如果案例不足 → 补充案例清单
2. 如果需要失败案例 → 用"失败案例标准格式"生成（见4.3节）
3. 如果框架需要调整 → 与用户确认后调整

---

#### 反思检查点3：深度与冲击力检查（完成75%进度时）

**触发时机**：完成第5-6章后（或总字数达到75%时）

**检查对象**：反常识洞察、失败案例细节、理论深度

**深度与冲击力清单**：

- [ ] **文章有没有"反常识"的冲击？**
  - 反常识标准：有"90%的人...""大部分人以为...但真相是..."
  - 检查方法：有没有挑战常规认知的观点？
  - 常见问题：全文都是常识，没有洞察
  
- [ ] **失败案例有没有"温水煮青蛙"的细节？**
  - 必须包含：时间线（如2019-2024）
  - 必须包含：决策点（如"第一次妥协"）
  - 必须包含：创始人反思（如"如果回到2020年..."）
  - 检查方法：案例是否只有结果，没有过程？
  
- [ ] **有没有"道"的阐释（不只是术）？**
  - 道的标准：解释WHY（底层逻辑），不只是HOW（操作步骤）
  - 检查方法：是否有"本质是什么""为什么有效"的章节？
  
- [ ] **读者能否产生"啊哈时刻"？**
  - 标准：让读者理解他们之前不理解的东西
  - 检查方法：有没有"原来是这样"的认知跃迁点？

**处理流程**：
1. 如果缺少反常识 → 增加"XX%的人回不到Y"等表述
2. 如果失败案例太简略 → 用"失败案例标准格式"重写（见4.3节）
3. 如果缺少"道" → 增加"底层逻辑"章节

**关键提醒**：
- 三次反思不是"选做"，而是"必做"
- 每次反思后，MUST检查标题是否需要同步调整
- 如果立场变了，标题必须跟着变
- 三次反思的目的是"在写作过程中迭代"，而不是"写完再改"

---

### 4.2 标题同步检查机制

标题是核心观点的浓缩，必须与文章立场完全一致。

**触发标题同步检查的情况**：
- 第一次反思调整了立场 → MUST检查标题
- 第二次反思扩展了框架 → MUST检查标题
- 第三次反思增加了"反常识"洞察 → MUST检查标题
- 任何全局性的观点调整 → MUST检查标题

**检查方法**：
- 标题能否一句话概括当前的核心观点？
- 标题有没有歧义或误导？
- 标题与核心金句是否呼应？

**如果标题需要调整**：
1. 提出3-5个新标题选项
2. 说明每个选项的优缺点
3. 用户确认后更新
4. 更新所有出现标题的地方（开头、结尾、YAML索引）

**常见错误**：
- 立场从"A错B对"调整为"不同阶段不同选择"，但标题还是"为什么要从A转B"
- 核心观点变成"B是杠杆，不是归宿"，但标题还是"从A到B的战略转移"

**记住**：标题不能滞后，必须与当前观点实时同步。

**【标题生成原则】——好标题必须同时满足三条，缺一不可**

条件一：与核心观点一致（防守线）
- 标题能否一句话概括核心观点？
- 有没有歧义或误导？

条件二：包含"被看见"的钩子（进攻线）
至少具备以下一种：
- 反常识陈述（"你的粉丝不是你的"）
- 具体真实人物或数字（"papi酱" / "8年"）
- 制造悖论或认知撕裂
- 直接戳痛点（"你缺的不是流量"）

条件三：读者3秒内感受到冲击（检验线）
- 检验方法：把标题给一个不了解内容的人看3秒
- 如果他们说"这是什么意思？" → 太难懂，改
- 如果他们说"然后呢？" → 好标题

禁止：
- 平铺型标题（只描述内容，如"财富公式的三个要素"）
- 说教型标题（让读者先思考才能感受冲击的）

记住：文章本身在讲"被看见"，标题就是文章第一次被看见的机会。
如果标题做不到吸引点击，文章里的道理再好，也是在自言自语。

### 4.3 关键内容前置生成原则

在实际写作中发现，某些类型的内容如果让执行工具生成，质量会严重下降。这些内容必须由思考工具（如Claude）前置生成完整文字，执行工具（如Cursor）直接复制使用。

**必须前置生成的内容类型**：

| 内容类型 | 为什么必须前置生成 | 质量标准 |
|---------|----------------|---------|
| **失败案例细节** | 需要"温水煮青蛙"的时间线、决策点、创始人反思 | 800-1000字 |
| **核心金句** | 需要提炼精髓（10-30字） | 至少3条 |
| **决策框架** | 需要逻辑严密+边界条件清晰 | 计算公式+示例 |
| **YAML索引** | 需要知识库上下文（doc_id、related_docs） | 完整字段 |
| **标题** | 必须与核心观点一致，经过三次反思确认 | 经反思确认 |

---

#### 失败案例的标准格式（800-1000字）

失败案例是"警示性"内容，需要让读者产生共鸣："这不就是我现在的状态吗？"

**标准格式**：

```markdown
**案例X：[公司名/项目名]的"[核心问题]"（[起始年]-[结束年]）**

**[起始年]（[阶段名称]）**：
- [背景：创始人是谁，想做什么]
- [初始想法：目标是什么]
- [MVP或第一步：具体行动]

**[转折年]（第一个转折/第一次妥协）**：
- [具体事件：发生了什么]
- [决策对话：还原当时的讨论]
  - 角色A（如技术）："[具体对话]"
  - 角色B（如销售）："[具体对话]"
  - 创始人："[具体决策]"
- **第一次妥协/错误决策**（关键节点，加粗强调）

**[中期年]（滑坡开始）**：
- [问题放大：第一次妥协导致的后果]
- [团队膨胀/现金流舒适：温水煮青蛙的状态]
- [创始人发现问题但改不动：为什么改不动？]

**[后期年]（无法回头）**：
- [市场变化：外部环境变了]
- [想改但来不及了：具体说明为什么晚了]
  - 技术锁定：代码库太复杂
  - 团队锁定：能力退化
  - 客户锁定：不接受改变

**[结束年]（结局）**：
- [最终结果：被收购/裁员/转型/倒闭]
- 创始人反思（必须有具体对话）：
  
  **"[金句1：对比反差，如'本来想做10亿的公司，现在只值8000万']"**
  
  **"[金句2：关键决策点反思，如'如果回到2020年，我一定会拒绝第一个定制需求']"**

**核心教训**：
- [教训1：普适性教训]
- **[教训2：强调关键决策点，如"第一次妥协是关键"]**
- [教训3：机制性教训，如"现金流是温水"]
```

**关键要素检查**：
- [ ] 有完整时间线（起始年-结束年）
- [ ] 有"第一次妥协"等关键决策点
- [ ] 有具体的决策对话（不是转述）
- [ ] 有"想改但改不动"的细节
- [ ] 有创始人的反思金句（带引号）
- [ ] 总字数800-1000字

### 6.1 标准写作框架


### 6.2 第一步：故事/案例开头（必须）

**目的**：用故事/案例吸引读者，引出核心问题

**要求**：
- 200-500字
- 必须与核心观点直接相关
- 有冲击力，让读者产生"这是我"的共鸣

**写作方式要求**：

**❌ 禁止的写法**：
- "李明"、"张三"、"小王"等明显虚构的名字
- 第三人称客观叙述（像作文）
- 铺垫太长，读者等不及要看观点

**✅ 推荐的写法**：
- "我有个朋友"、"我认识一个创业者"、"有个学员问我"
- 第一人称视角，增强真实感
- 直接对话（"他说XXX"）、具体细节（"做SaaS"、"2019年"）
- 即使虚构，也要让人觉得"这是真的"

**案例选择标准**：

| 优先级 | 标准 | 示例 |
|-------|------|------|
| **最高** | 反常识+简洁 | "噪音消失了，学生还是听不进去" |
| **次高** | 真实感+冲击力 | "我朋友差点把公司干倒闭" |
| **第三** | 完整过程 | "从0到1的完整创业故事" |

**案例选择优先级**（新增）：

| 优先级 | 标准 | 示例 |
|-------|------|------|
| **最高** | 真实案例（书籍、研究、亲身经历） | 《洞见》锯木厂案例 |
| **次高** | 虚构但有真实感（"我有个朋友"） | "我有个做SaaS的朋友" |
| **禁止** | 作文式虚构（"李明"、"张三"） | ❌ "李明第五次戒烟" |

**如何判断案例是否足够好？**
- ✅ 简洁（200-300字）
- ✅ 反常识（让人"啊？怎么可能？"）
- ✅ 直击本质（一句话说清洞察）
- ❌ 铺垫太长（500字+）
- ❌ 一看就是编的

**示例对比**：

**❌ 错误示例**（作文式虚构，铺垫太长）：
```markdown
李明是一个创业者。2019年，他第五次尝试戒烟。

前四次都失败了。第一次用意志力硬扛，三天后崩溃；
第二次用尼古丁贴片...（铺垫300字）

三个月后，李明成功戒烟。
```
**问题**：
- "李明"一看就是编的
- 铺垫太长，读者等不及
- 缺乏反常识冲击

**✅ 正确示例**（真实案例，简洁有力）：
```markdown
我读过一本书叫《洞见》，作者讲了自己在冥想静修时的经历。

那天，外面突然传来建筑工地的电锯噪音。刺耳、尖锐，持续不断。

正常情况下，任何人都会觉得烦躁，想关窗户、想逃避。

但作者做了一件反常识的事：他不是试图消除噪音，
而是用冥想审视自己对噪音的"判断"——

"为什么我觉得这声音'刺耳'？"
"'刺耳'是声音本身的属性，还是我大脑加工出来的？"

结果神奇的事情发生了：**当他剥离了"刺耳"这个判断后，
电锯噪音竟然听起来像音乐。**

噪音还在，但他不再痛苦。

这就是冥想的力量：**问题不在外部刺激，而在你给刺激贴的标签。**
```
**优势**：
- 真实（来自《洞见》）
- 简洁（250字）
- 反常识（"噪音像音乐"）
- 直击本质（"问题在判断"）

**反常识案例示例**：
```markdown
有个学校旁边有个锯木厂，每天噪音很大，学生根本无法专心听课。

校长想尽办法，终于说服锯木厂搬走了。

结果呢？学生还是听不进去课。

为什么？

因为问题不在外部噪音，而在学生已经养成的"分心习惯"。
外部环境变了，但内在反应模式没变。
```

**模板**：

```markdown
## 一、[故事标题]

[具体场景描述]

**问题/冲突**：[描述问题]

**结果**：[出人意料的结果]

**这就是本文要讨论的：[核心主题]**
```

**案例**：

```markdown
## 一、那个被"顺手"集成的功能

2018年，微软Teams宣布内置屏幕录制功能。消息一出，许多独立的录屏工具公司股价暴跌。

**为什么？**

因为用户发现：既然Teams已经有了录屏功能，为什么还要单独下载一个录屏工具？

**这就是初创企业面临的残酷现实：你的核心功能，可能只是大平台的"顺手"功能。**
```

## 七、写作风格要求

### 7.1 语气风格

**核心定位**：像一位带徒弟的CEO

**案例真实性金句**（松阳风格，写入写作原则）：
> 「编一个案例只需要5秒，搜一个真实案例需要5分钟，但说服力差了50倍。」

**特征**：
- 直白、真诚、有锋芒
- 敢泼冷水，也给路径
- 务实、带节奏感
- 不堆砌概念

**故事/案例开头的写作方式**（新增）：
- ❌ 禁止用"李明"、"张三"等作文式虚构名字
- ✅ 用"我有个朋友"、"我认识一个创业者"、"有个学员"
- ✅ 第一人称视角，增强真实感
- ✅ 直接对话（"他说XXX"），具体细节（"做SaaS"、"2019年"）
- ✅ 优先选择"反常识+简洁"的案例（如"噪音消失了，学生还是听不进去"）

**禁止**：
- 学术八股文
- 鸡汤式空洞
- 概念堆砌
- 过度修饰
- **作文式虚构故事**（新增）

### 7.2 语言要求

| 维度 | 要求 | 示例 |

## 十一、11项全局质检清单（交付前MUST执行）

写作完成后，对照以下清单进行全局质检。这是避免低质量输出的最后一道防线。

**质检时机**：
- 写作完成后
- 或每次重大修改后
- 或准备交付前

---

### 11.1 立场一致性（3项）

□ **1. 标题与核心观点一致吗？**
  - 检查方法：标题能否一句话概括核心观点？有没有歧义？
  - 常见问题："从C到B"暗示放弃C端，与"B端是路径，C端是目标"矛盾
  - 修复方法：提出3-5个标题选项，与用户确认

□ **2. 开场故事传递的观点正确吗？**
  - 检查方法：故事结尾的洞察与核心观点一致吗？
  - 常见问题：故事传递"C端错，B端对"，但正文说"不同阶段不同选择"
  - 修复方法：修改故事结尾的洞察部分

□ **3. 文章有没有自相矛盾的地方？**
  - 检查方法：全文搜索关键词，看前后表述是否一致
  - 常见问题：第二章说"B端天花板低"，第五章又说"B端规模化潜力大"
  - 修复方法：统一措辞，或增加"在X情况下"的限定

---

### 11.2 案例完整性（3项）

□ **4. 失败案例有时间线、决策点、创始人反思吗？**
  - 检查方法：案例是否包含年份、"第一次妥协"、"如果回到..."
  - 常见问题：只有结果（"某公司失败了"），没有过程细节
  - 修复方法：用"失败案例标准格式"（见4.3节）重写，800-1000字

□ **5. 主线案例在关键章节都出现了吗？**
  - 检查方法：搜索主线案例名（如"健康分"），是否在第1/3/6章出现
  - 常见问题：第一章讲健康分，后面再也没提
  - 修复方法：在关键章节增加主线案例的引用

□ **6. 案例与理论框架对应吗？**
  - 检查方法：每个理论点是否有案例支撑？
  - 常见问题：提出"三大陷阱"，但只有两个案例
  - 修复方法：补充案例，或调整框架

---

### 11.2.5 【内容质量】案例真实性（1项，新增）

□ **8. 案例真实性**
  - [ ] 开篇核心案例是否为真实可查的公开案例？
  - [ ] 或为松阳本人亲身经历？
  - [ ] 虚构案例是否已明确标注"示意"？

---

### 11.3 深度检查（2项）

□ **7. 有没有"反常识"的冲击？**
  - 检查方法：有没有"90%的人..."或"大部分人以为...但真相是..."
  - 常见问题：全文都是常识，没有洞察
  - 修复方法：增加"反常识"表述，如"90%转B端的人回不到C端"

□ **9. 有没有"道"的阐释（不只是术）？**
  - 检查方法：有没有解释WHY（底层逻辑），不只是HOW（操作步骤）
  - 常见问题：只告诉"怎么做"，不告诉"为什么这样做有效"
  - 修复方法：增加"底层逻辑"章节，解释机制

---

### 11.4 完整性检查（2项）

□ **10. YAML索引完整吗？**
  - 检查方法：doc_id、related_docs、query_patterns、training_questions都有吗？
  - 常见问题：字段缺失或doc_id编号错误
  - 修复方法：按照模板（见8.2节）补全所有字段

□ **11. 相关文档引用正确吗？**
   - 检查方法：引用的文档在知识库中存在吗？文档名准确吗？
   - 常见问题：引用了不存在的文档名，或文档名拼写错误
   - 修复方法：搜索验证文档存在

---

检查完成后，报告：
```

[如有问题，列出具体问题和修复建议]
```

1. 生成"修复指令"
2. 或标注问题，等待确认后修复
3. 修复后重新执行质检

## 十二、常见问题

### Q1：如何判断应该创建新文档还是补充现有文档？

**判断标准**：

| 情况 | 操作 |
|------|------|
| 现有文档已充分覆盖 | 不写 |
| 可以作为现有文档的一个章节 | 补充现有文档 |
| 是独立的主题，且内容>2000字 | 创建新文档 |
| 与现有文档有明显差异 | 创建新文档 |
